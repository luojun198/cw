# AI 接入可行性分析报告

面向系统：CW Finance 行政事业单位财务记账系统
分析日期：2026-05-12
分析范围：业务场景 × 技术可行性 × 落地优先级

---

## 一、现状与基础

### 1.1 已实现的 AI 功能

| 功能 | 文件 | 实现方式 | 状态 |
|------|------|---------|------|
| AI智能摘要 | `routes/voucherAi.ts` + `services/voucherEntry.ts` | 分录→摘要，调用 OpenAI 兼容 API | ✅ 可用 |
| AI异常检测 | `routes/reportAi.ts` | 单张凭证异常检测，调用 OpenAI 兼容 API | ✅ 可用 |
| AI配置 | `ai_config` 表 | 存 API URL/Key/Model/Enabled | ✅ 可用 |
| AI调用日志 | `ai_logs` 表 | 记录 input/output/tokens | ✅ 可用 |

### 1.2 现有基础设施

```
ai_config 表结构：
  id, account_set_id, provider, api_url, api_key, model, enabled, settings

ai_logs 表结构：
  id, account_set_id, user_id, type, input, output, tokens

调用方式：fetch(api_url) + Bearer token，同步输出
当前 Prompt 风格：角色设定 + 任务说明 + 结构化输出要求
```

### 1.3 关键约束

- **数据敏感性：** 行政事业单位财务数据不能随意发送到境外 API
- **响应时间：** 用户操作过程中调 AI 不能超过 2-3 秒，否则体验中断
- **网络限制：** 部分单位内网无法访问外网 API
- **成本控制：** 每月每单位 API 费用需控制在合理范围
- **用户水平：** 一线会计人员对 AI 的信任度低，需要"建议"而非"自动执行"

## 二、AI 接入场景分析

### 场景 A：凭证录入辅助（录入时效性最高）

#### A1. 智能科目推荐（价值最高，改动最小）

**当前痛点：**
会计录入凭证时，每一条分录都要手动选科目。行政事业单位有 500~1500 个科目，新会计经常选错。对经验丰富的会计也是重复劳动。

**AI 方案：**

```
用户输入摘要 → 前端请求 AI → 返回建议科目(带置信度) → 用户确认/修改
```

**Prompt 设计：**
```
系统：你是行政事业单位会计专家。根据业务摘要推荐最合适的会计科目。
      返回 JSON：{recommendations: [{code, name, reason, confidence}, ...]}

用户：购买办公用A4纸10箱，金额2000元
AI：{recommendations: [
  {code:"500101", name:"办公费", reason:"购买办公用品", confidence:0.95},
  {code:"500199", name:"其他商品和服务支出", reason:"办公用品可归入此类", confidence:0.30}
]}
```

**技术实现：**
| 方案 | 延迟 | 成本 | 离线可用 | 说明 |
|------|------|------|---------|------|
| 实时API调用 | ~1-3s | 高 | 否 | 用户每次选科目前等待 |
| 本地关键词匹配 | <10ms | 0 | 是 | 预建"关键词→推荐科目"映射表，覆盖80%场景 |
| 混合方案 | <10ms / ~1s | 低 | 部分 | 先查本地表，命中率<70%时回退到API |

**推荐：混合方案。** 从历史凭证中提取"摘要→科目"对应关系建立本地索引。本地命中率低于阈值时再调用 API。

**UI 交互：**
```
[摘要输入框: 购买办公用A4纸10箱]
↓ 自动请求（500ms防抖）
[科目下拉框] 旁边出现 AI 推荐标记✨
  ┌─────────────────────────────┐
  │ ⭐ 500101 办公费 (95%)      │  ← AI推荐居首
  │    500199 其他商品和服务(30%)│
  │    ...全部科目              │
  └─────────────────────────────┘
```

**改动范围：**
- 后端：`POST /api/voucher/ai/recommend-account`（可复用现有 ai/summary 框架）
- 前端：`VoucherEntryForm.vue` 中科目选择器增加 AI 推荐逻辑
- 新增：`services/accountLocalIndex.ts` 本地关键词→科目索引

**预计投入：** 后端 1 天 + 前端 1 天

#### A2. 附件 OCR → 自动生成凭证草稿（高级功能）

**痛点：** 单位的报销凭证需要会计逐张录入发票信息。

**AI 方案：**

```
扫描件/照片 → OCR提取(金额/日期/对方/摘要) → AI判断业务类型 → 生成分录草稿
```

**依赖：**
- 前端：图片上传组件 + 预览
- OCR 引擎：Tesseract.js（免费，纯前端）或 百度/腾讯 OCR API（收费，精度高）
- AI 推理：将 OCR 文本输入 LLM 判断业务类型和科目

**实现提示：**
```
OCR文本：
  发票代码：xxx
  购买方：XX财政局
  商品：A4复印纸 10箱 单价200 金额2000.00
  开票日期：2026-05-10

LLM输出：
  业务类型：办公用品采购
  推荐分录：[{科目:500101, 方向:借, 金额:2000}, {科目:100201, 方向:贷, 金额:2000}]
```

**成本估算：** 每次 OCR + LLM 调用约 0.05~0.1 元，每张发票约 0.1 元。

### 场景 B：凭证审核辅助（降低审计风险）

#### B1. 增强版异常检测

**当前限制：** 现有 `reportAi.ts` 只对单张凭证做检测，没有横向对比。

**改进方向：**

| 检测类型 | 当前能力 | 增强方案 |
|---------|---------|---------|
| 单张异常 | ✅ 已实现 | 完善检测规则库 |
| 同类对比 | ❌ | 批量发送同类凭证给 AI，标出金额异常偏大的 |
| 连号检查 | ❌ | 查询本期凭证号序列，找出断号 |
| 突击花钱 | ❌ | 月初/月末金额对比，标出月末异动 |
| 同一经手人 | ❌ | 统计某制单人本月金额异常 |

**Prompt 示例（同类对比）：**
```
系统：你是一个财务审计助手。以下是本月第2期所有"办公费"科目的凭证
      （共15张，金额范围100~50000元），请标出金额明显偏大的异常凭证。

凭证列表：
  记-021 5月10日 办公用品 3,000
  记-035 5月12日 打印机墨盒 500
  记-052 5月15日 采购办公家具 48,000  ← 异常
  记-068 5月18日 文具 200
  ...

AI输出：
  异常凭证：记-052（5月15日 48,000元）
  原因：金额为同类凭证中位数的16倍，建议核实采购审批流程是否完整
```

**改动范围：**
- 新增 `POST /api/voucher/ai/batch-anomaly` 批量异常检测接口
- 前端审核页面增加"AI批量检查"按钮
- 每次调用约消耗 500~2000 tokens，成本 ~0.01 元/次

#### B2. 政府会计制度合规检查（行政事业单位专属）

**痛点：** 政府会计制度与企业会计制度完全不同，许多基层单位容易用错科目。

**常见错误：**
- 用企业会计科目代替政府会计科目（如"管理费用"替代"单位管理费用"）
- 固定资产入账漏记"固定资产累计折旧"
- "公共基础设施"和"固定资产"混淆

**AI 方案：**
```
凭证分录 → AI对照政府会计制度科目表 → 返回合规检查结果
```

**实现依赖：**
- 需要构建一个"政府会计制度科目约束知识库"（约 200 条规则）
- 可硬编码为 JSON 规则，也可让 AI 基于角色知识判断
- 混合方案：规则引擎兜底 + AI 补充判断

**规则示例：**
```json
{
  "code": "1601",
  "name": "固定资产",
  "paired_accounts": ["1602 固定资产累计折旧"],
  "required_aux": ["资产类别", "管理部门"],
  "forbidden_with": ["5001"],  // 不能与业务活动费用出现在同一分录
  "warn_rules": [
    "单笔>100万需政府采购审批流程附件",
    "月发生额>预算30%需预警"
  ]
}
```

### 场景 C：预算执行监控（行政事业单位最需要）

**背景：** 企业看利润，行政事业单位看预算执行率。《预算法》要求单位严格控制预算，超预算支出是严重违规。

#### C1. 实时预算看板

**数据源：**
- 预算数：从系统参数或单独导入（建议新增 `budget_items` 表）
- 执行数：从 `account_balances` / `voucher_entries` 实时计算

**AI 输出样例：**
```
截至5月20日，贵单位预算执行情况：

  科目         年度预算   已执行     执行率    剩余
  500101 办公费  500,000  325,000   65%     175,000
  500102 印刷费  200,000  180,000   90% 🔴   20,000
  500103 咨询费  100,000   15,000   15% 🟢   85,000
  500104 水电费  300,000  280,000   93% 🔴   20,000

⚠ 预警：印刷费和水电费执行率已超90%
💡 建议：本年度剩下的7个月，建议印刷费控制在每月3000以内
```

**新增表结构建议：**

```sql
CREATE TABLE budget_items (
```

**实现方式：**
- 预算看板页面：前端定时刷新（或被动查询）

### 场景 D：财务分析报告自动生成（减轻月末负担）

#### D1. 一键生成财务分析说明

**当前痛点：** 每月末会计需要写财务分析说明，内容包括：
- 本月收入支出概况
- 与上月/上年同期的对比
- 异常波动的解释
- 资产负债结构分析
- 预算执行情况

**AI 方案：**

```typescript
// cronjob: 每月1号凌晨2点执行
async function generateMonthlyReport(accountSetId, year, period) {
  // 1. 收集数据
  const balanceSheet = await getBalanceSheetData(...)
  const incomeData = await getIncomeStatementData(...)
  const budgetData = await getBudgetData(...)
  const prevPeriodData = await getPrevPeriodData(...)

  // 2. 构建 Prompt 上下文
  const prompt = buildAnalysisPrompt({
    current: { balanceSheet, incomeData, budgetData },
    previous: prevPeriodData,
    period: `${year}年${period}期`,
  })

  // 3. 调用 AI
  const analysis = await callAI(prompt, { system: "你是行政事业单位财务分析师..." })

  // 4. 存入数据库
  saveReport(accountSetId, year, period, analysis)

  // 5. 通知用户
  notifyUser("本月财务分析报告已生成")
}
```

**Prompt 结构：**
```
系统：你是行政事业单位财务分析师。根据提供的财务数据生成专业分析报告。
      报告结构：
      1. 总体概况（200字以内）
      2. 收入分析（与上月/上年同期对比）
      3. 支出分析（重点标注波动>20%的科目）
      4. 预算执行分析
      5. 资产负债分析
      6. 建议与关注事项

用户输入：
  本期数据：[JSON]
  上期数据：[JSON]
  上年同期数据：[JSON]

AI输出：（纯中文Markdown）
```

**存储方案：**
```sql
CREATE TABLE ai_reports (
  id TEXT PRIMARY KEY,
  account_set_id TEXT NOT NULL,
  year INTEGER NOT NULL,
  period INTEGER NOT NULL,
  report_type TEXT NOT NULL CHECK(type IN ('monthly', 'quarterly', 'annual')),
  content TEXT NOT NULL,
  tokens_used INTEGER,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  UNIQUE(account_set_id, year, period, report_type)
);
```

**成本估算：** 每次月度报告约 2000~5000 tokens，成本约 0.05 元/份

### 场景 E：自然语言查询（用户体验提升）

#### E1. NL → API 参数翻译

**目标：** 用户用自然语言提问，系统自动转换为结构化查询并返回结果。

**现有查询方式（传统）：**
```
会计操作步骤：
1. 点开"账簿查询"菜单
2. 选择"明细账"
3. 从500个科目的下拉框找到"500101"
4. 选日期范围（上月1号~上月最后一天）
5. 点查询
6. 看表格数据
```

**AI 增强后：**
```
用户输入：「查一下上个月500101办公费的明细」

系统内部流程：
1. AI分析意图 → { query_type: "ledger_detail", 
   account_code: "500101", time_range: "last_month" }
2. 翻译为 API 调用 → /api/ledger/detail?account_id=xxx&start_date=...&end_date=...
3. 返回数据 + AI 摘要 → "上月办公费共发生12笔，合计32,500元"

前端展示：表格数据 + AI文字说明
```

**技术方案：**

```
用户输入
    ↓
轻量级意图分类（正则 + 关键词，<1ms）
    ├─ 命中 → 直接映射到API参数
    └─ 未命中 → 调用LLM（1~2s）
                   ↓
            解析JSON输出 → 调用API → 返回结果
```

**意图分类规则（可本地处理 80% 查询）：**

| 用户输入模式 | 意图 | 示例 |
|-------------|------|------|
| `查[科目]的明细账` | ledger_detail | "查500101的明细账" |
| `[科目]的总账` | ledger_general | "业务活动费用的总账" |
| `[月]的余额表` | ledger_balance | "5月的余额表" |
| `[科目]余额` | account_balance | "银行存款还有多少" |
| `[月]的资产负债表` | report_bs | "上个月资产负债表" |
| `[科目]比[科目]` | compare | "办公费比上月多多少" |
| `哪些凭证超过[金额]` | voucher_filter | "哪些凭证超过10万" |

**前端交互设计：**
```
┌──────────────────────────────────────────────┐
│ 🔍 输入问题...                         [发送] │
├──────────────────────────────────────────────┤
│                                              │
│  例：查上个月500101的明细账                    │
│  例：哪些凭证超过10万还没审核                   │

│                                              │
```

### 场景 F：系统运维辅助（开发者福利）

#### F1. 智能建账向导

**痛点：** 每新装一套系统，需要导入科目体系、配置凭证类型、设置报表格式。

**AI 方案：** 用户选择单位类型（小学/中学/医院/乡镇政府/财政局等），AI 自动推荐：
- 科目体系配置
- 凭证类型（记账/收款/付款/转账）
- 预设结转规则
- 常用报表模板
- 辅助核算类别（功能分类/项目/部门等）

**实现方式：** 不需要实时 AI，预先生成模板 JSON 即可。

#### F2. 错误分析与恢复建议

**场景：** 过账失败、凭证号冲突、约束违反等。

**AI 方案：** 捕获异常后，将错误信息 + 当前上下文发送给 AI，返回修复建议。

```
错误：UNIQUE constraint failed: vouchers(account_set_id, year, period, voucher_type_id, voucher_no)
AI建议：凭证号冲突，凭证号"记-023"已被占用。建议操作：
  1. 查看"记-023"是否是该凭证本身（重复保存）
  2. 如果不是，请点击"重新排号"自动编号
```

## 三、技术架构建议

### 3.1 分层调用架构

```
前端页面
  │
  ├─ 低延迟需求（<100ms）→ 本地规则引擎 / 关键词匹配
  │   科目推荐本地索引、意图分类、简单规则检查
  │
  ├─ 中延迟需求（1~3s）→ 实时 API 调用
  │   科目推荐（本地未命中时）、异常检测、单凭证审核
  │
  └─ 高延迟需求（可接受分钟级）→ 异步任务 / cronjob
      月度报告生成、批量异常扫描、预算执行报告
```

### 3.2 现有接口复用

当前 `POST /api/voucher/ai/summary` 接口可以扩展为通用 AI 接口：

```typescript
// 改造成通用 AI 助手接口
POST /api/ai/ask
Body: {
  type: 'recommend-account' | 'anomaly-check' | 'analysis' | 'query',
  input: { ... },
  context: { year, period, accountSetId }  // 自动带上系统上下文
}
Response: {
  code: 0,
  data: { result: ..., tokens: ..., model: ... }
}
```

### 3.3 数据隐私方案

| 数据等级 | 示例 | 处理方式 |
|---------|------|---------|
| 低敏感 | 科目名称、凭证类型 | 可发送到云端 API |
| 中敏感 | 摘要、金额范围 | 脱敏后发送（金额模糊化） |
| 高敏感 | 单位全称、银行账号、人员姓名 | 不得发送到外部 API，使用本地模型 |

**本地模型方案：** 对于高敏感单位，可部署本地 LLM（如 deepseek-v4-pro 或 Qwen3），使用 Ollama 或 vLLM 提供服务。
