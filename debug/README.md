# debug / 修复日志

本目录用于记录代码评审之后所有 BUG 修复的详细过程，便于后续查阅与回滚。

## 文件结构

- `修复日志.md` ：按时间倒序记录每一次修复（最新在最上）
- `代码评审报告.md` ：原始评审报告（如未拆出请见对话）
- `fixes/FIX-{ID}-{slug}.md` ：单个修复的详细技术档案（必要时单独拆出）

## 修复记录写入规则

> **强制规则：每完成一项 BUG 修复（无论大小），必须在 `debug/修复日志.md` 文件顶部追加一条记录。提交代码前完成记录写入。**

每条记录使用如下模板：

```markdown
## [YYYY-MM-DD HH:mm] FIX-{编号} {标题}

- **严重级别**：P0 / P1 / P2 / P3
- **问题来源**：代码评审报告 §{对应条目编号}
- **影响范围**：哪些功能/页面/数据会受影响
- **根因**：一两句话说清楚为什么错
- **修改文件**：
  - `path/to/file.ts` ：改了什么
  - `path/to/another.ts` ：改了什么
- **新增/删除字段或表**：（如涉及 DB 迁移）
- **数据修复**：是否需要历史数据回算、回算脚本路径
- **测试**：跑了哪些单测 / 手工验证步骤
- **风险与回滚**：上线前注意事项、回滚方法
- **相关 issue / 任务**：链接或编号
```

## 编号约定

- `FIX-001`, `FIX-002`, ...，按完成时间递增。
- 同一 BUG 修复跨多个 PR 时，编号保留不变，新增 `-a/-b` 后缀。

## 何时不需要写入

- 纯文档调整、注释修正、格式化（无逻辑变更）
- 临时调试代码、未上库的实验性改动

## 与项目优化日志的关系

`debug/修复日志.md` 关注**单个 BUG 的根因和回归依据**，面向开发与回滚；
项目根目录 `优化日志.md` 关注**功能/特性维度**的演进，面向用户与运维。
两者并存，互相引用。
