# 2026-04-12 政府会计科目初始化设计

## 目标

以 `2023年最新版政府会计准则制度（科目表+自动结转模板）.md` 为唯一口径，重建项目中的标准会计科目模板，使系统支持：

- 财务会计科目 1-5 类
- 预算会计科目 6-8 类
- 当前账套执行强制重建后，`/base/account` 页面能展示正确、完整的树形科目体系

本次工作聚焦“科目体系初始化/重建”，不扩展无关业务功能；但若现有自动结转、期初余额或科目展示因新科目口径直接受影响，则做必要兼容修正。

## 现状与问题

当前系统已有标准科目初始化逻辑，核心入口位于：

- `server/src/scripts/seedAccounts.ts`
- `server/src/scripts/rebuildAccounts.ts`
- `server/src/index.ts`

当前 `seedAccounts.ts` 中的标准科目模板与用户提供的 2023 文档存在明显不一致，至少包括：

- 文档口径为 `1021 其他货币资金`，现模板中存在 `1012 其他货币资金`
- 文档中 `1811`、`1821` 等编码对应名称与现模板不一致
- 当前模板更多是历史整理版，不是严格按此次文档全量对齐的结果

因此不能通过页面手工少量修补解决，必须把“系统标准科目源”改正确。

## 设计原则

1. **单一标准源**：系统初始化与重建应共享同一份标准科目模板，避免首次初始化和后续重建结果不一致。
2. **文档优先**：以用户提供的 2023 文档作为本次科目编码、名称、方向和结转相关口径的唯一依据。
3. **结构兼容现系统**：在遵循文档的前提下，保留系统现有字段能力，例如层级、父子关系、现金/银行标志、辅助核算配置、删除权限等。
4. **显式破坏性操作**：本次重建是明确的人工执行动作，不应在页面加载或服务启动时对已有账套自动触发。

## 方案比较

### 方案 A（推荐）
直接重构 `server/src/scripts/seedAccounts.ts` 中的标准科目模板，并继续复用现有初始化与重建入口。

**优点：**
- 初始化默认账套与后续重建共用同一套标准数据
- 长期维护成本最低
- 最符合“按文档设置正确完整科目”的目标

**缺点：**
- 需要系统性梳理现有模板并整体校正，改动面较大

### 方案 B
保留现有模板，新增一套“2023 完整版科目模板”并单独导入。

**优点：**
- 对旧逻辑侵入较小

**缺点：**
- 项目中会并存两套标准源
- 后续初始化、重建、校验容易分叉

### 方案 C
只写一次性数据库导入脚本，不改 `seedAccounts.ts`。

**优点：**
- 一次性见效快

**缺点：**
- 新环境初始化仍会得到旧模板
- 重建、校验和默认账套初始化仍然不一致

### 推荐结论
采用 **方案 A**。

## 目标实现结构

### 1. 标准科目模板重构
核心修改点为 `server/src/scripts/seedAccounts.ts`。

将现有 `ACCOUNT_TEMPLATES` 重构为按文档组织的全量模板，覆盖：

- 财务会计 1-5 类
- 预算会计 6-8 类

每条科目至少保留以下信息：

- `code`
- `name`
- `direction`
- `level`
- 父子关系
- 必要时的 `is_cash` / `is_bank`
- 必要时的辅助核算配置

当文档只提供一级科目时，一级科目必须严格对齐；若系统现有二级/三级明细与文档口径不冲突，可在后续实现中决定是否保留、精简或同步重构，但整体结构必须服务于“全量、正确、可展示”。

### 2. 初始化与重建入口复用
保留当前入口关系：

- 默认账套初始化：`server/src/index.ts`
- 强制重建脚本：`server/src/scripts/rebuildAccounts.ts`

这样可以保证：

- 新环境首次启动获得正确科目
- 老环境执行重建获得同一套正确科目

### 3. 前端页面以验证为主
`client/src/views/base/Account.vue` 当前负责树表展示。只要后端科目层级、父子关系和字段结构正确，页面应可直接展示。

本次仅在必要时修正两类问题：

- 树层级或父子关系导致的展示异常
- 页面对新数据字段的不兼容问题

不对页面做与本目标无关的大改。

### 4. 校验脚本同步更新
同步修正 `server/src/scripts/verifyAccounts.ts`，使其可验证新模板结果，例如：

- 科目总量与层级分布
- 关键一级科目是否存在
- 关键预算会计科目是否存在
- 方向与关键属性是否符合预期

## 数据策略

本次以**当前账套**为目标账套，并执行**强制重建**：

1. 删除当前账套下现有 `accounts`
2. 按文档全量重建 1-8 类科目

用户已明确确认：即使当前账套已有凭证或期初余额引用旧科目，也仍然执行强制重建。

## 风险与处理

### 主要风险
强制重建会导致旧科目 ID 失效，因此凡是依赖旧 `account_id` 的数据都可能受影响，至少包括：

- `voucher_entries.account_id`
- `init_balances.account_id`
- 以及任何基于科目 ID 的查询、报表、余额计算逻辑

### 风险处理策略
1. 实现前先检查当前账套中是否已有凭证、期初余额等引用科目的业务数据，并将现状报告给用户。
2. 即使发现引用，仍按用户已确认的策略执行强制重建。
3. 不把重建动作做成隐式自动行为，而是保留为明确的初始化/重建动作，降低误触发风险。
4. 若重建后发现直接受影响的系统功能（如 `/base/account` 展示、科目查询、期初余额加载）出现兼容问题，应在本次范围内做必要修正。

## 验收标准

完成后应满足以下条件：

1. `/base/account` 页面可正常打开并展示完整树形科目
2. 科目范围覆盖文档中的 1-8 类
3. 关键编码与名称正确，例如：
   - `1001` 库存现金
   - `1002` 银行存款
   - `1011` 零余额账户用款额度
   - `1201` 财政应返还额度
   - `3301` 本期盈余
   - `6001` 财政拨款预算收入
   - `8101` 财政拨款结转
4. 文档中明确体现的 2023 口径修正已反映到模板中
5. 重建脚本与校验脚本可用
6. `/base/account` 页面展示、搜索、树结构无明显异常

## 非目标

本次不包含以下内容，除非为兼容新科目体系所必需：

- 重做凭证录入业务规则
- 全面重构自动结转逻辑
- 重做报表口径
- 扩展新的辅助核算类别或复杂初始化向导
