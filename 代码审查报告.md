# 代码审查报告

审查日期：2026-05-12
审查范围：server/src/ + client/src/
项目：CW Finance 行政事业单位财务记账系统

---

## 一、严重 Bug（必须修复）

### B1. 总账/余额表父科目金额汇总重复计算

**文件：** `server/src/services/ledgerQuery.ts`（buildLedgerGeneralQuery, 第86-247行）
**文件：** `server/src/routes/ledger.ts`（第22-191行, 第291-427行）

**问题：**
SQL查询先用 `NOT EXISTS (SELECT 1 FROM accounts child WHERE child.parent_id = a.id AND child.is_enabled = 1)` 排除了所有父科目（只查末级科目），再通过 `UNION ALL` 把父科目加回来但金额全为0。然后在前端路由层（`ledger.ts`）通过 `getParentCodes()` 重新计算父科目编码并手动汇总子科目金额到父科目。

**后果：**
1. 父科目的期初余额被硬编码为0，之后再通过子科目汇总加回去，如果子科目数据不完整，父科目余额就会错
2. 父科目汇总在JS端进行，如果某个子科目因为过滤条件被排除，父科目汇总就不完整
3. 余额表（`/balance`接口）和总账（`/general`接口）的父科目补充逻辑几乎完全重复（约80行重复代码）

**建议：** 统一用SQL一次性计算所有科目的余额（包括父科目按子科目汇总），不需要在JS端再做二次汇总。

---

### B2. 过账时辅助核算项 aux_item_id 只取了 dept_id

**文件：** `server/src/services/voucherPosting.ts`（第86行, 第136行）
```typescript
const auxItemId = (entry as any).dept_id || ''
```

**问题：**
过账时 `account_balances` 表的 `aux_item_id` 只取了 `dept_id`，完全忽略了 `project_id`、`supplier_id`、`person_id`、`func_class_id` 以及自定义辅助核算的 `aux_data` 里的项目。

**后果：**
1. 如果一个科目同时挂接了项目和部门辅助核算，系统只会按部门汇总，项目的余额数据会混乱
2. 唯一约束 `UNIQUE(account_set_id, account_id, year, period, aux_item_id)` 会导致不同项目但同部门的数据被合并
3. 辅助核算账簿查询结果必然错误

**建议：** aux_item_id 应该是一个复合值（如 `dept_id:project_id:supplier_id`），或者使用 `aux_data` 中的 JSON 内容作为唯一标识。

---

### B3. 月末日期硬编码为31日

**文件：** `server/src/services/ledgerQuery.ts`（第324行）
```typescript
const periodEnd = `${y}-${String(month).padStart(2, '0')}-31`
```

**问题：** 2月、4月、6月、9月、11月使用31日作为月末日期。在SQLite中 `'2026-02-31'` 会被转为NULL比较，导致这些月份的数据查询不到或结果错误。

**建议：** 使用 `date('YYYY-MM-01', '+1 month', '-1 day')` 在SQL中生成正确的月末日期。

---

### B4. 结转生成凭证可能借贷不平

**文件：** `server/src/services/autoTransfer.ts`（第372-392行）
```typescript
const fromDirection = childAccount.direction === 'credit' ? 'debit' : 'credit'
```

**问题：** 对于父科目下的多个子科目，方向反转逻辑只处理了转出方，没有对应的转入方处理逻辑。如果 `buildEntriesFromTransferItems` 中的 `targetItems` 和 `sourceItems` 配对关系处理不完善，会导致结转分录借贷方金额不符。

**建议：** 在 `buildAutoTransferEntries` 函数中对生成的 entries 做一次借贷平衡校验，如果不平则抛出明确错误。

---

### B5. FTS5 全文搜索 rowid 与主键不一致

**文件：** `server/src/services/voucherQuery.ts`（第108-133行）

**问题：**
FTS5 虚拟表返回的 `voucher_id` 是 rowid（整数类型），但 `vouchers` 表的主键 `id` 是 TEXT(UUID)。代码用 `v.rowid IN (...)` 来过滤，但 FTS5 表与主表的 rowid 可能不一致（尤其是经过迁移重建表之后）。

```typescript
conditions.push(`v.rowid IN (${placeholders})`)
```

而且 FTS5 搜索结果的 `voucher_id` 被定义为 `number` 类型（`as Array<{ voucher_id: number }>`），但UUID是字符串。

**建议：** 使用FTS5的 `content=` 外部内容表功能，或者使用 `content_rowid=` 保持 rowid 一致性。最简单的修复是改用主键 id 关联而不是 rowid。

## 二、重要问题（建议修复）

### I1. 期间关闭检查缺失

**文件：** `server/src/routes/voucherPosting.ts`（第21-83行）
**文件：** `server/src/services/voucherPosting.ts`

**问题：** 过账操作完全没有检查 `period_closing` 表。用户可以在已月结的期间继续过账或反过账，导致已封账的数据被篡改。

**建议：** 过账前校验 `isPeriodClosed(getPeriodClosingRecord(...))`

---

### I2. reportBalance.ts getBalance方向混乱的风险

**文件：** `server/src/services/reportBalance.ts`（第14-49行）

**问题：**
```typescript
const dir = init?.direction || periodBal?.direction || 'debit'
```
`init_balances` 和 `account_balances` 的 direction 都可能返回多行（`GROUP BY a.direction`），但 `.get()` 只取第一行。如果科目体系中有借贷方向混用的子科目，这里会得到错误的 direction。

另外，`getBalance` 函数用于资产负债表计算，`catch` 块返回0会静默隐藏所有SQL错误，导致报表数据凭空消失。

**建议：** 使用统一方向（从 `accounts` 表查），`try-catch` 至少输出warning日志。

---

### I3. 余额表父科目汇总可能存在双重累加

**文件：** `server/src/routes/ledger.ts`（第56-72行、第303-317行）

**问题：** 先对SQL结果用 `account_code` 做 `deduplicatedMap` 合并（因为SQL中末级科目+父科目UNION可能返回重复），然后又补充缺失父科目并做子->父汇总。如果同一个子科目数据在UNION后有多行，先合并再汇总的值可能被重复计算。

**建议：** 在SQL层面用 `SUM` + `GROUP BY` 一次性归并，消除JS端的手动去重。

---

### I4. 反过账回退逻辑不完整

**文件：** `server/src/services/voucherPosting.ts`（第138-158行）

**问题：** 反过账时 `revertBalanceExact` 如果没匹配到行（changes===0），会调用 `revertBalanceFallback` 尝试用 `aux_item_id IS NULL OR aux_item_id=''` 条件回退。但如果同一个科目在同期间同 aux_item_id 下有多笔过账记录，fallback 只会减去最后一个分录的金额，留下错误余额。

**建议：** 反过账时使用更精确的匹配条件，或每次过账保存明细日志以便反过账时精确回滚。

---

### I5. 资产负债表使用 LIKE 匹配科目编码可能越界

**文件：** `server/src/services/reportBalance.ts`（第24行、第35行）

**问题：** 使用 `a.code LIKE ?` 配合 `${accountCode}%` 来匹配科目范围，但 `group codes` 中定义的是前4位科目编码（如'1001'），`LIKE '1001%'` 会匹配到 '1001001'（如果存在的话），这些可能并不是期望的子科目。

**建议：** 根据 `account_code_lengths` 配置精确截取编码长度，或使用 `>=`/`<` 范围匹配。

---

### I6. 凭证审核钩子缺失校验

**文件：** `server/src/routes/voucherAudit.ts`
**文件：** `server/src/services/voucherEntry.ts`

**问题：** 审核操作没有校验凭证的分录是否完整（如借方贷方是否都有科目、金额是否都大于0），也没有检查凭证日期是否在已关闭的会计期间。

**建议：** 审核前补充校验逻辑。

## 三、代码质量问题

### Q1. 大量重复代码

**范围：** 整个项目

**问题：**
- `ledger.ts` 路由中 `/general`、`/balance`、`/general-ledger` 三个端点有几乎一样的父科目补充逻辑（`getParentCodes` + `accountMap` + `missingParents`），共约250行重复代码
- `buildWhereClause` 工具函数在 `ledgerQuery.ts` 中定义，但在 `voucherQuery.ts` 和部分路由中又重新写了类似的逻辑
- 各账簿查询的 `statusCondition` 字符串重复出现5次以上

**建议：** 提取通用工具函数，统一处理父科目补充、状态过滤、去重逻辑。

---

### Q2. console.log 调试日志未清理

**文件：** `server/src/routes/ledger.ts`（第74行、第93-95行、第100-103行、第110行、第127-129行、第134行、第366行、第388行、第393行）
**文件：** `client/src/composables/useSearchMemory.ts`

**问题：** 大量 `console.log` 在生产环境会暴露数据结构并影响性能。

**建议：** 改用 `log.debug` 或统一清理。

---

### Q3. SQL字符串硬拼接存在注入风险

**文件：** `server/src/services/ledgerQuery.ts`（多处）

**问题：** 多处使用模板字符串拼接日期和状态条件到SQL中，如：
```typescript
`AND v.voucher_date >= '${periodStart}'`
`AND v.voucher_date <= '${periodEnd}'`
`AND ${statusCondition}`
```

虽然目前这些值是在服务器端生成的，但风格不一致，且日后维护时容易引入注入风险。

**建议：** 全部改用参数化查询。

---

### Q4. 凭证编号重新排号没有事务保护

**文件：** `server/src/routes/voucher.ts`（第182-196行）

**问题：** `updateAll` 虽然是 `transaction()`，但如果在执行过程中有其他操作插入了新凭证，会导致编号冲突（唯一约束违反）。而且没有使用 `SELECT ... FOR UPDATE` 或排它锁。

**建议：** 排号操作应该先锁表（`BEGIN IMMEDIATE`）。

---

### Q5. 操作日志同步写入影响性能

**文件：** `server/src/middleware/log.ts`

**问题：** `operationLog` 装饰的每个请求都会同步写一次 `operation_logs` 表。高频操作（如查看凭证列表）会产生大量日志，占用数据库I/O。

**建议：** 使用内存队列批量写入，或仅记录关键写操作（增/删/改），查询类操作不记录。

---

### Q6. 前端提交格式与后端存储格式不一致

**文件：** `client/src/composables/useVoucherForm.ts`
**文件：** `server/src/services/voucherEntry.ts`

**问题：** 前端用 `debit_amount`/`credit_amount` 两个字段表示借贷方，后端用一个 `amount` + `direction` 字段，每次提交都需要转换，增加出错几率。建议统一。

---

### Q7. 类型定义分散且不一致

**问题：** 
- 前端 `VoucherEntry` 接口和后端 `VoucherEntryInput` 接口字段不同
- `AuxItemLike`、`AuxCategoryLike` 等类型在多个文件中重复定义
- `SqlParam` 类型（`string | number`）不能覆盖所有SQL参数类型（如 `null`、`boolean`、`bigint`）

**建议：** 使用 shared-types 包统一管理类型定义，前后端共享。

---

### Q8. 前端 Entry.vue 文件体积过大

**文件：** `client/src/views/voucher/Entry.vue`（约1000行）
**文件：** `server/src/routes/ledger.ts`（约1284行）

**建议：** 拆分到多个独立组件/路由文件。

## 四、数据库设计问题

### D1. 缺少关键索引

当前缺失的索引：
| 表名 | 建议索引 | 理由 |
|-------|---------|------|
| `voucher_entries` | `(account_set_id, account_id)` | 账簿查询核心条件 |
| `init_balances` | `(account_set_id, account_id, year)` | 期初余额计算 |
| `operation_logs` | `(account_set_id, created_at)` | 日志查询 |
| `voucher_entries` | `(voucher_id, seq)` | 已存在（migration 5） |
| `account_balances` | `(account_set_id, account_id, year, period)` | 余额表查询 |

### D2. 数字金额使用 REAL 类型

**问题：** 所有金额字段都使用了 `REAL` 类型，在大量累加时可能出现浮点精度误差（0.1+0.2 问题）。

**建议：** 对于财务系统应全部使用整数（单位：分），或者 SQLite 中使用 `NUMERIC(18,2)` 但 SQLite 内部仍按 REAL 处理。更安全的方案是前端后端统一用 `BigInt`/`分` 存储。

## 五、安全性问题

### S1. JWT Token 过期时间8小时

**文件：** `server/src/middleware/auth.ts`（第65行）
```typescript
return jwt.sign(payload, getJwtSecret(), { expiresIn: '8h' })
```

**问题：** 财务系统通常有合规要求（等保三级），8小时过长，且没有刷新Token机制。

**建议：** Token 有效期设为2小时，配合 refreshToken 实现无感刷新。

### S2. 密码没有强制复杂度校验

**文件：** `server/src/routes/auth.ts`

**问题：** 注册/修改密码时没有复杂度要求（大小写+数字+特殊字符），也没有检查密码是否与用户名相同。
