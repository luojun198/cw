# 压力性能审查报告

审查场景：每月 10,000+ 张凭证，每张约 5 条分录（50,000+ 条分录/月）
数据库：SQLite（better-sqlite3，WAL模式）
同步API，单连接

---

## 🔴 一级风险（万条级必卡）

### P1. 凭证列表查询：LEFT JOIN 导致 5x 中间行膨胀

**文件：** `server/src/services/voucherQuery.ts`（第212-218行）
```sql
SELECT COUNT(DISTINCT v.id) as count
FROM vouchers v
LEFT JOIN voucher_entries ve ON v.id=ve.voucher_id  -- ← 问题
WHERE ...

SELECT DISTINCT v.*, vt.name as voucher_type_name
FROM vouchers v
LEFT JOIN voucher_types vt ON v.voucher_type_id = vt.id
LEFT JOIN voucher_entries ve ON v.id = ve.voucher_id  -- ← 问题
WHERE ...
```

**问题：** `LEFT JOIN voucher_entries` 在没有任何 entry 级过滤的条件下被加入查询。1万张凭证 × 5条分录 = 5万条中间行，然后 `DISTINCT` 被迫对这5万行全量排序去重。

COUNT 查询同样被拖慢——5万行去重后COUNT(*)。

**压力估算：** 基础查询（无关键字、无科目过滤）在1万张凭证下约需 200-500ms，而这应该只需 5-10ms。

**修复方案：** 仅当 `accountId/accountIds/keyword/auxItems/auxFields` 等 entry 级过滤存在时才 LEFT JOIN voucher_entries，否则直接从 vouchers 表查。去掉 `DISTINCT`（主键已经唯一）。

### P2. 总账查询：13个关联子查询 / 科目，O(N²) 级复杂度

**文件：** `server/src/services/ledgerQuery.ts`（第86-207行）

**问题：** `buildLedgerGeneralQuery` 对每个科目执行 13 个独立的关联子查询：
- `init_balance` → 2个子查询（init_balances + 凭证累计）
- `current_debit` → 1个
- `current_credit` → 1个
- `year_debit` → 1个
- `year_credit` → 1个
- `end_balance` → 2个子查询 + 派生逻辑
- 再加 5 个派生计算的副本

500个科目 × 13个子查询 = **6,500次子查询执行**。每次子查询都 `JOIN vouchers v ON v.id = ve.voucher_id`，等价于扫描 voucher_entries 表 6,500 次。

**压力估算：** 50,000条分录下，每次子查询约 1-5ms，总账接口单次响应时间可能高达 **6.5秒~30秒**。

**修复方案：** 将这13个子查询合并为 1-2 个聚合查询，用 `CASE WHEN direction='debit' THEN amount ELSE 0 END` 一次性取出所有方向的数据，再用 `SUM(CASE WHEN ...)` 在应用层或SQL层分拆。

### P3. 总分类账（带月度汇总）：24个子查询/科目

**文件：** `server/src/services/ledgerQuery.ts`（第278-318行）

**问题：** 对1-12月的每个月，分别执行借方/贷方两个子查询：
```
for month = 1 to 12:
  monthN_debit → 1个子查询
  monthN_credit → 1个子查询
```
500科目 × 24 = **12,000次子查询**。每个子查询的SQL结构几乎一样，只是日期范围不同。

**压力估算：** ~12,000次表扫描，单次请求可能 **15秒以上**。

**修复方案：** 改用 `GROUP BY strftime('%m', v.voucher_date)` 按月分组，一次查出所有月份的数据，然后在应用层拆到12个月份的列中。

### P4. 资产负债表：N+1 逐科目查询

**文件：** `server/src/services/reportBalance.ts`（第7-49行）
**文件：** `server/src/routes/reportBalanceSheet.ts`（第96-123行）

**问题：** `calcItem(codes)` 对每个科目代码单独调用 `getBalance()`，而 `getBalance()` 内部又执行 2 次独立SQL查询（init_balances + account_balances）。

资产负债表约 30-50 个科目行，加上备抵科目的 `calcNetValueItem`（多科目相减），总共 **约 100 次 SQL 查询**。每次查询都使用 `LIKE` 匹配，索引效率远不如等值查询。

**压力估算：** 50,000分录 × 科目 LIKE匹配 × 100次 ≈ 3~10秒

**修复方案：** 
1. 将资产负债表的所有科目余额一次性用 `WHERE a.code IN (...)` 查出来放到 Map 中，应用层做加减
2. 改用等值匹配 + 编码范围（`>= <`）代替 LIKE

---

### P5. FTS5全文搜索：全量结果集放入 IN 子句

**文件：** `server/src/services/voucherQuery.ts`（第108-133行）

**问题：** FTS5 搜索把全部匹配的 voucher_id 拉到内存做成 Set，然后拼成超长的 `IN` 子句：
```typescript
const matchedVoucherIds = new Set<number>()
voucherFtsResults.forEach(r => matchedVoucherIds.add(r.voucher_id))
entryFtsResults.forEach(r => matchedVoucherIds.add(r.voucher_id))
// ...
const placeholders = voucherIdList.map(() => '?').join(',')
conditions.push(`v.rowid IN (${placeholders})`)
params.push(...voucherIdList) // 5000个参数
```

如果关键词匹配 5000 张凭证，IN 子句包含 5000 个参数，SQLite 需要为每个参数构建查询计划。另外 `v.rowid` 是 SQLite 内部 rowid，但 FTS5 的 `voucher_id` 可能是 UUID 的下标，两者不一定一致。

**压力估算：** 5000个参数的 IN 子句构建约 100-300ms，加上 FTS5 搜索本身，单次搜索约 500ms-2秒。

**修复方案：** 
1. FTS5 使用 `content=` 外部内容表保持 rowid 一致性
2. 如果不是必须用全文搜索，改为 `LIKE` + 索引扫描（对财务摘要搜索够用）
3. 如果必须用 FTS5，限制搜索结果上限（如 `LIMIT 200`）

### P6. 分页深翻页：OFFSET 扫描丢弃大量行

**影响范围：** 所有使用 `LIMIT ? OFFSET ?` 的查询

**问题：** 翻到第 200 页（page=200, pageSize=50），`OFFSET=9950`。SQLite 需要扫描全部 10,000 行，构造结果集，然后丢掉前 9,950 行。

**压力估算：** 首页（第1页）约 10-50ms，第 200 页约 500ms-2秒。

**修复方案：** 对于凭证列表，使用游标分页（keyset pagination）：
```sql
WHERE (voucher_date, voucher_no) > (?, ?)  -- 记录上一页的最后一条
ORDER BY voucher_date, voucher_no
LIMIT ?
```
不需要 OFFSET，永远只扫需要的行数。

对于账簿报表（总账/余额表），不存在翻页需求（一次性展示全部），改为不分页。

## 🟡 二级风险（高负载可能卡）

### P7. 过账事务逐条 UPSERT 在月底爆发

**文件：** `server/src/services/voucherPosting.ts`（第106-136行）

**问题：** 每周期的第一条凭证过账时，`INSERT ... ON CONFLICT DO UPDATE` 在 account_balances 为空时快速INSERT。但到月底当 account_balances 表已有数万行时，每次 UPDATE 的冲突检测需要查找唯一索引。

实际影响较小，因为唯一索引查找是 O(log N)，数万行下每次约 0.01ms。

**修复方案：** 无需修改。但建议在每月初 `DELETE FROM account_balances WHERE account_set_id=? AND year=? AND period=?` 后重建索引（`REINDEX`）。

### P8. voucher_entries 缺少 direction + 金额覆盖索引

**当前索引：** `idx_voucher_entries_account ON voucher_entries(account_set_id, account_id)`

**问题：** 所有总账子查询都包含 `AND ve.direction = 'debit'`、`AND ve.direction = 'credit'` 以及 `SUM(ve.amount)`。但当前索引只覆盖到 `(account_set_id, account_id)`，SQLite 必须回表读 `amount` 和 `direction` 字段。

建议索引：
```sql
CREATE INDEX idx_voucher_entries_balance
  ON voucher_entries(account_set_id, account_id, direction, amount);
```
这是一个覆盖索引，SQLite 可以直接从索引中拿到 direction 和 amount，无需回表。

### P9. 操作日志同步写入拖慢响应

**文件：** `server/src/middleware/log.ts`

**问题：** 每次关键操作（凭证录入/审核/过账）都同步写一次 `operation_logs` 表。单次写入很快（~1ms），但批量操作时（如批量审核100张凭证）要写100条日志。

如果 10,000 张凭证每月有 5 次操作（录入+修改+审核+过账+反过账）= 50,000条日志，日志表膨胀到数十万行。`INSERT` 在无索引的大表上会变慢。

**修复方案：** 日志写入改为异步队列，或使用内存 buffer 每 5 秒批量 flush。

### P10. 初始化函数在每次启动时执行但不幂等

**文件：** `server/src/index.ts`（第162-189行）

**问题：** `initDatabase()` 每次启动都执行完整的 `schema.sql`（包含 `CREATE TABLE IF NOT EXISTS` 和所有索引）。虽然 `IF NOT EXISTS` 不会重复创建，但 500行SQL的解析+执行需要时间。`seedRoles()` 使用 `INSERT OR IGNORE` 但每次都全量查询 `SELECT * FROM roles`。

服务重启时这些函数都会执行，单次影响不大（~50ms），但如果频繁重启（开发模式watch）会累积。

## 🟢 低风险/建议项

| 编号 | 问题 | 影响 |
|------|------|------|
| P11 | `uuidv4()` 主键在 5000万行下碎片化，`ORDER BY` 性能劣化 | 长期（1亿行+） |
| P12 | 所有金额用 REAL 类型，SUM 累计浮点误差累积 | 建议改为整数（分） |
| P13 | `getPeriodSum` 和 `getPeriodRangeSum` 用 `period<=?` 扫描所有历史月份 | 年末查询时可能有数百毫秒延迟 |
| P14 | SQLite WAL 文件在密集写入时膨胀 | 建议定时执行 `PRAGMA wal_checkpoint(TRUNCATE)` |
| P15 | 前端的 `useSearchMemory` localStorage 序列化/反序列化 | 仅影响前端，单次 <1ms |
