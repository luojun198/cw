## 一、严重 BUG（可能导致数据错误或功能故障）

### BUG-01: 凭证过账时辅助核算ID取值错误 — 导致余额表数据污染
**文件**: [voucherPosting.ts:106](server/src/services/voucherPosting.ts#L106)  
**严重程度**: 🔴 严重

```typescript
const auxItemId = (entry as any).dept_id || ''
```

过账时 `aux_item_id` 硬编码取 `dept_id`，而非从分录的实际辅助核算字段中动态获取。`account_balances` 表的唯一约束是 `(account_set_id, account_id, year, period, aux_item_id)`，这意味着：
- 所有分录（无论有无辅助核算）都按 `dept_id` 写入余额
- 如果分录有 `project_id` 但无 `dept_id`，余额记录的 `aux_item_id` 为空字符串，导致不同辅助维度的余额混在一起
- 反过账时同样按 `dept_id` 匹配，可能匹配不到正确的余额记录

**修复建议**: 从 `aux_data` JSON 中正确解析当前辅助核算维度对应的 `aux_item_id`，或根据科目配置动态确定。

### BUG-02: 反过账余额回退 fallback 逻辑可能扣除错误记录
**文件**: [voucherPosting.ts:129-138](server/src/services/voucherPosting.ts#L129-L138)  
**严重程度**: 🔴 严重

```typescript
if (!result.changes || result.changes === 0) {
  revertBalanceFallback.run(...)
}
```

当精确匹配 `aux_item_id` 失败时，fallback 到 `aux_item_id IS NULL OR aux_item_id=''` 的记录。这可能匹配到本不该扣减的余额记录，导致余额数据错误。如果过账时写入了空字符串 `aux_item_id`，而实际应该匹配带值的记录，此 fallback 会扣错行。

**修复建议**: 反过账应严格按与过账一致的键值匹配，匹配失败应报错而非静默 fallback。

### BUG-03: 过账写入 `end_balance` 始终为 0 — 资产负债表金额错误
**文件**: [voucherPosting.ts:100-104](server/src/services/voucherPosting.ts#L100-L104)  
**严重程度**: 🔴 严重

```sql
INSERT INTO account_balances (..., end_balance, end_debit, end_credit, ...)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, 0, 0, 0, ?)
ON CONFLICT(...) DO UPDATE SET
  current_debit = current_debit + excluded.current_debit,
  current_credit = current_credit + excluded.current_credit
```

`end_balance`、`end_debit`、`end_credit` 在 INSERT 时设为 0，ON CONFLICT 时也不更新。这意味着 `end_balance` 永远为 0 或初始值，资产负债表、科目余额表等依赖 `end_balance` 的报表数据全部不正确。

**修复建议**: 过账后需重新计算 `end_balance = init_balance + current_debit - current_credit`（借方科目）或 `end_balance = init_balance + current_credit - current_debit`（贷方科目）。

### BUG-04: 过账完整性检查余额查询依赖 `end_balance`，始终返回 0

**文件**: [voucher.ts:527-537](server/src/routes/voucher.ts#L527-L537)

```typescript
  const allEntries = db.prepare(...).all(...)
  const entriesMap = buildVoucherEntriesMap(allEntries)  // ← 内层重新声明！遮蔽了外层
```

**文件**: [auth.ts:211](server/src/routes/auth.ts#L211)

### SEC-03: AI配置接口返回 API Key 明文
**文件**: [reportAi.ts:77-82](server/src/routes/reportAi.ts#L77-L82)  
**严重程度**: 🔴 严重

```typescript
router.get('/ai/config', (req: AuthRequest, res) => {
  const config = db.prepare(...).get(req.accountSetId)
```

**修复建议**: 返回时对 `api_key` 做脱敏处理（如只返回后4位），或仅在有修改需求时通过 PUT 接口传递。

### SEC-08: 前端 Token 存储在 localStorage — XSS 攻击风险

## 三、设计不合理

### DES-01: 路由层包含大量业务逻辑 — 违反分层架构
**涉及文件**: 几乎所有 routes/*.ts  
**严重程度**: 🟡 中

路由处理器中直接编写 SQL 查询、业务校验、事务逻辑。例如 `voucher.ts` 超过 800 行，包含凭证创建、修改、删除、附件管理等全部逻辑。仅有 `voucherPosting.ts`、`voucherQuery.ts`、`accountService.ts` 等少数服务被抽离。

**修复建议**: 将业务逻辑下沉到 Service 层，路由层仅负责参数校验、调用 Service、格式化响应。

---

### DES-02: 凭证编号生成存在并发竞争
**文件**: [voucherEntry.ts:239-270](server/src/services/voucherEntry.ts#L239-L270)  
**严重程度**: 🟠 高

`getNextVoucherNo` 先查询 MAX 编号再生成新编号，两步之间无锁。并发请求可能获取相同编号，导致凭证号冲突。

**修复建议**: 在数据库事务内执行查询和插入，或使用 SQLite 的 `BEGIN EXCLUSIVE` 事务。

---

### DES-03: 账套删除禁用外键检查 — 数据一致性风险
**文件**: [system.ts:152-172](server/src/routes/system.ts#L152-L172)  
**严重程度**: 🟠 高

```typescript
const deleteAll = db.transaction(() => {
  db.pragma('foreign_keys = OFF')
  // ... 逐表 DELETE
  db.pragma('foreign_keys = ON')
})
```

临时关闭外键检查进行级联删除，如果中途失败，数据库将处于不一致状态。且 SQLite 的 `PRAGMA foreign_keys` 在事务内修改的行为在某些版本中不完全可靠。

**修复建议**: 利用 `ON DELETE CASCADE` 外键约束自动级联删除，或按依赖顺序逆序删除。

---

### DES-04: 报表任务管理使用内存存储 — 不可靠
**文件**: [reportTaskManager.ts:21](server/src/services/reportTaskManager.ts#L21)  
**严重程度**: 🟡 中

```typescript
const tasks = new Map<string, ReportTask>()
```

任务状态存储在进程内存中，服务器重启后全部丢失。多实例部署时也无法共享状态。

**修复建议**: 对于单实例部署可接受，但应添加注释说明限制。长期应考虑使用 SQLite 或 Redis 持久化。

---

### DES-05: 验证码存储在 system_params 表 — 不合理
**文件**: [auth.ts:89-93](server/src/routes/auth.ts#L89-L93)  
**严重程度**: 🟡 中

验证码数据存入 `system_params` 表，该表设计用途是存储系统配置参数，非临时数据。验证码验证成功后仅删除对应记录，但过期未使用的验证码永不清理，导致表数据膨胀。

**修复建议**: 使用 Redis 或独立的 `captcha_store` 表，并添加定时清理机制。

---

### DES-06: 科目余额表总账查询极度低效
**文件**: [ledgerQuery.ts:56-150](server/src/services/ledgerQuery.ts#L56-L150)  
**严重程度**: 🟠 高

每个科目的 6 个数值字段（init_balance, current_debit, current_credit, year_debit, year_credit, end_balance）各使用一个相关子查询，导致 N×6 次子查询。加上 UNION ALL 补充父科目，性能随数据量增长急剧恶化。

**修复建议**: 使用 CTE + JOIN 一次性计算所有科目的余额，避免相关子查询。

---

### DES-07: 总账路由层做大量后处理 — 应在 SQL 或 Service 层完成
**文件**: [ledger.ts:31-152](server/src/routes/ledger.ts#L31-L152)  
**严重程度**: 🟡 中

总账查询返回数据后，在路由层做了：去重合并 → 补充父科目 → 子科目金额汇总到父科目 → 筛选过滤。这些逻辑应在 Service 层或通过改进 SQL 完成。

---

### DES-08: `db` 类型定义过于宽泛
**涉及文件**: 多个 service 文件  
**严重程度**: 🟡 中

多个 Service 函数接受 `db: any` 或手动定义的最小接口（如 `{ prepare: (sql: string) => { all: (...args: any[]) => any[] } }`），而非使用 `Database.Database`。这丧失了类型安全，且无法利用 IDE 自动补全。

**修复建议**: 统一使用 `Database.Database` 类型，仅在单元测试中使用 mock 接口。

---

### DES-09: `X-AccountSet-Id` 请求头未使用
**文件**: [request.ts:16-18](client/src/api/request.ts#L16-L18)  
**严重程度**: 🟢 低

前端每个请求都附带 `X-AccountSet-Id` 头，但服务端 `authMiddleware` 只从 JWT 读取 `accountSetId`，完全忽略此头部。这是无用代码，还可能造成误导。

**修复建议**: 移除前端发送 `X-AccountSet-Id` 的逻辑，或服务端改为从头部读取（需增加安全校验）。

### STD-09: `db/index.ts` 中的 `ensureXxx` 函数破坏迁移系统完整性
**文件**: [db/index.ts](server/src/db/index.ts)  
**严重程度**: 🟡 中

使用 `window.location.href` 做硬跳转会丢失所有 Vue/Pinia 状态。应使用 `router.push('/login')`。

路由配置中有 `meta: { requiresAuth: true }`，但未见 `beforeEach` 导航守卫实现。未登录用户可直接访问需认证的页面。

### FE-03: 登录响应类型处理不安全
**文件**: [user.ts:31-38](client/src/stores/user.ts#L31-L38)  
**严重程度**: 🟢 低

```typescript
const res = (await login(form)) as LoginResponse
userInfo.value = res.user as UserInfo
```

使用 `as` 强制类型断言，如果后端返回结构变化不会报错，而是运行时崩溃。

## 六、性能问题

### PERF-01: 总账查询使用 7+ 个相关子查询
**文件**: [ledgerQuery.ts](server/src/services/ledgerQuery.ts)  
**严重程度**: 🟠 高

如 DES-06 所述，每个科目行产生 7 个相关子查询，100 个科目就是 700 个子查询。应重构为 JOIN 聚合查询。

---

### PERF-02: 凭证列表查询 LEFT JOIN 分录表但仅用于关键词搜索
**文件**: [voucherQuery.ts:92-106](server/src/services/voucherQuery.ts#L92-L106)  
**严重程度**: 🟡 中

```sql
SELECT COUNT(DISTINCT v.id) as count 
FROM vouchers v LEFT JOIN voucher_entries ve ON v.id=ve.voucher_id
```

仅当有 `keyword` 搜索时才需要 JOIN `voucher_entries`，但无关键词时也做了 JOIN，增加了不必要的开销。

---

### PERF-03: 批量操作逐条处理而非批量SQL
**文件**: [voucherAudit.ts:46-65](server/src/routes/voucherAudit.ts#L46-L65)  
**严重程度**: 🟡 中

批量审核/反过账在循环中逐条查询+更新，每次都是独立的 SQL 语句。应使用 `WHERE id IN (...)` 批量查询，然后在事务中批量更新。
