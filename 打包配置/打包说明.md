# CW Finance - Windows 零依赖部署打包说明

## 方案原理

```
开发机 (Node 25)                    目标服务器 (零依赖)
┌─────────────────┐                ┌─────────────────────┐
│  构建            │                │                     │
│  ├─ Vite 构建前端 │                │  deploy-final/      │
│  ├─ esbuild 打包  │   复制整个文件夹 │  ├─ node/          │
│  └─ npm install   │ ──────────────→│  ├─ server/        │
│    (用 Node 20)   │                │  ├─ client/dist/   │
│                   │                │  ├─ data/          │
│                   │                │  ├─ uploads/       │
│                   │                │  ├─ 模版/          │
│                   │                │  └─ start.bat      │
└─────────────────┘                └─────────────────────┘
```

## 为什么不用 PKG

| 对比项         | PKG (单 exe)         | Portable Node.js |
| -------------- | -------------------- | ---------------- |
| 原生模块兼容   | 差（固定 Node 版本） | 好（版本可控）   |
| better-sqlite3 | 需要手动匹配 ABI     | 自动下载预编译   |
| 调试难度       | 高                   | 低（直接看日志） |
| 包大小         | ~50MB                | ~107MB           |
| 维护成本       | 高                   | 低               |

## 核心兼容性问题

better-sqlite3 是 C++ 原生模块，`.node` 二进制文件必须与 Node.js ABI 版本严格匹配：

| better-sqlite3 | Node.js 版本 | ABI |
| -------------- | ------------ | --- |
| v12.8.0        | 25.x         | 141 |
| v12.x          | 22.x         | 127 |
| v12.x          | 20.x         | 115 |

**关键步骤**：必须用与开发环境相同版本的 portable Node.js（当前为 25.9.0）执行 `npm install`，这样 better-sqlite3 会自动下载对应 ABI 的预编译二进制。

**备选方案**（当网络无法访问 GitHub 下载预编译二进制时）：

1. 在开发机先用系统 Node.js 构建好 better-sqlite3：`npm rebuild better-sqlite3`
2. 将编译好的 `.node` 文件复制到部署包的对应位置

## 打包步骤

### 推荐：构建 + 压缩 zip（发给客户）

根目录双击：

```
build-and-package.bat
```

或：

```
打包配置\一键打包.bat
```

会先执行 `build-deploy.bat` 生成最新 `deploy-final`，再压缩到 `deploy-output\cw-deploy-v*-b*.zip`。

仅压缩已有目录（已构建过）可双击根目录 `压缩部署包.bat`，或：

```
压缩部署包.bat skip-build
```

**不要**只运行 `scripts\compress-deploy-package.ps1` 而不先构建——该脚本历史上仅压缩已有 `deploy-final`，容易把旧程序打成 zip。

**不要**在 Windows 资源管理器中右键 `deploy-final` →「发送到压缩文件夹」。`deploy-final` 体积大且含 `node_modules`，Explorer 会在 `%LocalAppData%\Temp\` 下建临时目录，易出现「位置不可用」；请一律用上述脚本，输出在 `deploy-output\`。

### 方式一：仅生成 deploy-final 目录

```
build-deploy.bat
```

### 方式二：开发中快速同步程序（不重下 Node、不重装依赖）

在已有完整 `deploy-final` 的前提下：

```
scripts\sync-deploy-artifacts.bat
```

仅重新构建并覆盖 `server\bundle.cjs` 与 `client\dist`，并写入 `BUILD_STAMP.json`。

### 核对部署包是否最新

**登录页版本号（推荐）**

打包并启动后打开登录页（`http://localhost:3005`），卡片底部会显示例如：

`v1.0.0 · 构建 202605271713 · b2619e8`

- 与 `deploy-final\BUILD_STAMP.json` 中的 `version` / `build` / `gitCommit` 应一致
- 若出现 **「前后端版本不一致」**，说明 `client/dist` 与 `server/bundle.cjs` 未同步更新，请重新执行 `build-deploy.bat` 或 `scripts\sync-deploy-artifacts.bat`

构建时由 `scripts/generate-build-info.cjs` 生成 `client/public/version.json`，随 Vite 复制到 `client/dist/version.json`。

**文件核对**

查看 `deploy-final\BUILD_STAMP.json`：

- `inSyncWithProjectDist` 应为 `true`
- `build` 字段应与登录页「构建」号一致
- `clientIndexHash` 应与当前 `client\dist\index.html` 一致

也可对比 `deploy-final\client\dist\index.html` 与项目 `client\dist\index.html` 中引用的 `assets/index-*.js` 文件名是否相同。

### 手动打包

```batch
:: 1. 构建前端（跳过类型检查，使用 workspace 命令）
npm run build:deploy --workspace=client

:: 2. 构建后端 bundle
cd server
node scripts\buildBundle.mjs
cd ..

:: 3. 准备目录
mkdir deploy-final\server deploy-final\client deploy-final\data deploy-final\uploads

:: 4. 下载解压 Node.js 25.9.0
::    https://nodejs.org/dist/v25.9.0/node-v25.9.0-win-x64.zip
::    解压后重命名为 deploy-final\node\

:: 5. 用 portable Node.js 安装依赖（关键！）
cd deploy-final\server
copy ..\..\server\package.json .
set PATH=..\node;%PATH%
npm install --omit=dev
cd ..\..

:: 6. 复制文件
copy server\dist\bundle.cjs deploy-final\server\
copy server\src\db\schema.sql deploy-final\server\schema.sql
xcopy client\dist deploy-final\client\dist\ /E /I /Y
xcopy 模版 deploy-final\模版\ /E /I /Y
```

## 部署包目录结构

```
deploy-final/
├── node/                           # Node.js 25.9.0 portable (~102MB)
│   ├── node.exe
│   └── npm, npx, ...
├── server/
│   ├── bundle.cjs                  # 后端代码 (~1.67MB)
│   ├── schema.sql                  # 数据库结构
│   ├── .env                        # 环境变量配置
│   └── node_modules/               # 生产依赖 (~22MB)
│       └── better-sqlite3/
│           └── build/Release/
│               └── better_sqlite3.node  # 预编译二进制
├── client/
│   └── dist/                       # 前端静态文件 (~2.4MB)
│       ├── index.html
│       └── assets/
├── data/                           # 数据库（运行时创建或预置）
│   └── finance.db
├── uploads/                        # 上传文件目录
├── 模版/                           # ACD 模板文件
│   ├── *.acd                       # 会计准则模板
│   ├── 新企业会计准则/              # 新企业会计准则模板（含 Excel 报表）
│   └── 测试账套/                    # 测试账套 ACD 文件
├── QR.png                          # 二维码图片
└── start.bat                       # 启动脚本
```

## start.bat 注意事项

- 自动打开浏览器访问 `http://localhost:3005`
- 使用**纯英文**，避免 GBK/UTF-8 编码乱码
- Windows Server 2012 默认代码页 936 (GBK)
- `chcp 65001` 在某些系统上会导致 bat 文件内中文乱码
- 服务端 Node.js 日志中的中文不受影响（Node.js 内部 UTF-8）

## start.bat 内容

```batch
@echo off
title CW Finance Server
echo ========================================
echo   CW Finance - Starting...
echo ========================================
echo.
echo   Opening: http://localhost:3005
echo   Login  : admin / admin123
echo.
echo   Press Ctrl+C to stop server
echo ========================================
echo.
set "PATH=%~dp0node;%PATH%"
start http://localhost:3005
node server\bundle.cjs
pause
```

## .env 配置

部署时自动创建 `server/.env`，默认内容：

```
JWT_SECRET=your-secret-key-change-this-in-production
```

生产环境部署时，请修改 `JWT_SECRET` 为强随机字符串。

## 前后端技术栈

### 前端 (client/)

- Vue 3 + TypeScript
- Vite 5 构建
- Element Plus UI 组件库
- Pinia 状态管理
- ECharts 图表
- Vue Router 路由
- axios HTTP 客户端

### 后端 (server/)

- Express.js 框架
- TypeScript 源码，esbuild 打包为 CJS bundle
- better-sqlite3 数据库
- JWT 认证
- Winston 日志
- Multer 文件上传

### 外部模块（external，不在 bundle 中）

- better-sqlite3（C++ 原生模块）
- iconv-lite（编码转换）
- exceljs（Excel 读写）

## 升级指南

如果需要更换 Node.js 版本：

1. 修改 `build-deploy.bat` 或 `打包配置\一键打包.bat` 中的 `NODE_VER` 变量
2. 检查 better-sqlite3 是否支持新版本
3. 重新运行打包脚本

如果需要更换 better-sqlite3 版本：

1. 修改 `server/package.json` 中的 `better-sqlite3` 版本
2. 确保与 Node.js 版本兼容（见上方兼容表）
3. 重新运行打包脚本
