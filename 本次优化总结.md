## 一、核心成果

### 1. 浮点精度转换（完成）

**问题：** 财务系统使用 REAL 类型存储金额，存在浮点精度误差（0.1 + 0.2 = 0.30000000000000004）

**解决方案：** 采用渐进式整数分存储方案
- 数据库添加 `amount_cents INTEGER` 字段（Migration 14）
- 应用层优先使用整数计算，避免浮点误差累积
- 新凭证双写 REAL 和 INTEGER 字段，保持向后兼容

**修改文件：**
- `server/src/db/migrationList.ts` - 数据库迁移
- `server/src/utils/amountUtils.ts` - 金额转换工具（新建）
- `server/src/routes/voucher.ts` - 凭证录入双写
- `server/src/services/voucherEntry.ts` - 构建 payload 添加整数字段
- `server/src/services/voucherPosting.ts` - 过账逻辑优先使用整数
- `server/src/services/autoTransfer.ts` - 自动结转支持整数

**数据验证：**
- 迁移 4876 条历史记录，0 条不一致
- 单元测试全部通过（36/36）

---

### 2. FTS5 全文搜索修复（完成）

**问题：** FTS5 触发器使用错误语法，导致凭证删除/修改失败

**解决方案：** 使用 FTS5 content table 模式的正确语法
- DELETE 触发器：`INSERT INTO fts_table(fts_table) VALUES('delete', old.rowid)`
- UPDATE 触发器：先删除旧记录，再插入新记录

**修改文件：**
- `server/data/cw_finance.db` - 应用 fix-fts5-triggers-v3.sql

**验证结果：**
- 成功创建 6 个触发器（vouchers 和 voucher_entries 各 3 个）
- 凭证 CRUD 操作正常
- FTS5 索引正确同步

---

### 3. 防抖节流统一管理（完成）

**问题：** 前端存在手动实现的防抖逻辑，代码分散

**解决方案：** 使用统一的 `useDebounceThrottle.ts` composable
- 移除手动 setTimeout 实现
- 改用 `useDebounceFn(fn, delay)` 统一管理
- 自动处理清理，防止内存泄漏

**修改文件：**
- `client/src/components/voucher/VoucherEntryForm.vue`

**优化效果：**
- 代码减少 5 行
- 统一管理，提升可维护性

**浮点精度转换采用双写方案：**
- ✅ 新数据同时写入 REAL 和 INTEGER 字段
- ✅ 应用层优先使用 INTEGER 计算
- ✅ 历史数据通过 Migration 迁移
- ✅ 保持向后兼容，降低风险

**优势：**

- 避免大规模数据库改造

### 2. 完善的工具函数库

```typescript
yuanToCents(yuan)      // 元转分

useDebouncedRequest(fn, delay)     // 防抖请求（带取消）
useThrottledRequest(fn, delay)     // 节流请求
```
