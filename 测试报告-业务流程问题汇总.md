# 财务记账系统业务流程测试报告

## 测试范围

通过代码审查方式，对以下业务流程进行了全面分析：

1. 账套管理
2. 科目管理
3. 凭证录入
4. 凭证修改/删除
5. 审核/反审核
6. 过账/反过账
7. 结转损益
8. 报表生成

## 发现的问题及优化建议

### 一、账套管理流程

#### 问题 1：新建账套后缺少初始化引导

**位置**: `server/src/routes/system.ts:22-77`

**问题描述**:

- 创建账套后只返回成功消息，没有引导用户进行后续必要操作
- 用户不知道需要：导入科目模板、设置期初余额、配置系统参数等

**优化建议**:

```typescript
// 返回时增加初始化步骤提示
res.json({
  code: 0,
  message: '创建成功，已自动创建管理员账号 admin/admin123',
  data: {
    id,
    nextSteps: [
      { step: 1, title: '导入科目模板', required: true },
      { step: 2, title: '设置期初余额', required: true },
      { step: 3, title: '配置系统参数', required: false },
    ],
  },
})
```

#### 问题 2：账套切换后状态不清晰

**位置**: `server/src/routes/auth.ts:214-251`

**问题描述**:

- 切换账套后，前端可能缓存了旧账套的数据
- 没有强制刷新机制，可能导致数据混乱

**优化建议**:

- 切换账套后返回 `needRefresh: true` 标志
- 前端收到后清空所有缓存并重新加载

#### 问题 4：科目删除检查不够全面

**位置**: `server/src/services/accountService.ts:280-297`

**问题描述**:

- 只检查了凭证使用，没有检查：
  - 是否有子科目
  - 是否有期初余额
  - 是否在报表模板中使用

**优化建议**:

```typescript
// 增加完整的删除前检查
deleteAccount(id: string): void {
  const account = this.getAccountById(id)
  if (!account) throw new Error('科目不存在')
  if (!account.allow_delete) throw new Error('系统内置科目无法删除')

  // 检查子科目
  const children = this.db.prepare(
    'SELECT COUNT(*) as count FROM accounts WHERE parent_id = ?'
  ).get(id) as any
  if (children?.count > 0) {
    throw new Error('该科目有子科目，请先删除子科目')
  }

  // 检查期初余额
  const initBalance = this.db.prepare(
    'SELECT COUNT(*) as count FROM init_balances WHERE account_id = ?'
  ).get(id) as any
  if (initBalance?.count > 0) {
    throw new Error('该科目有期初余额，请先清除期初余额')
  }

  // 检查凭证使用
  const used = this.db.prepare(
    'SELECT COUNT(*) as count FROM voucher_entries WHERE account_id = ?'
  ).get(id) as any
  if (used?.count > 0) {
    throw new Error('该科目已被凭证使用，无法删除')
  }

  this.db.prepare('DELETE FROM accounts WHERE id = ?').run(id)
}
```

**问题描述**:

- 用户不知道凭证号是如何生成的
- 如果想插入凭证到特定位置，操作复杂

**优化建议**:

- 提供 `/vouchers/next-voucher-no` 接口（已存在）
- 前端在录入前显示"预计凭证号：记-001"
- 提供"插入凭证"功能，允许指定凭证号位置

#### 问题 7：凭证分录最少2条的限制不合理

**位置**: `server/src/services/voucherEntry.ts:34-39`

**问题描述**:

```typescript
if (!Array.isArray(entries) || entries.length < 2) {
  return '凭证日期和分录不能为空'
}
```

- 错误消息不准确，应该是"至少需要2条分录"
- 某些特殊业务可能需要多借多贷，但没有上限提示

**优化建议**:

```typescript
export function validateVoucherEntryCount(entries: unknown) {
  if (!Array.isArray(entries)) {
    return '分录格式错误'
  }
  if (entries.length < 2) {
    return '凭证至少需要2条分录（一借一贷）'
  }
  if (entries.length > 100) {
    return '凭证分录不能超过100条'
  }
  return null
}
```

### 四、凭证修改/删除流程

#### 问题 9：已审核凭证无法修改，但没有快捷反审核

**位置**: `server/src/routes/voucher.ts:537-578`

**问题描述**:

- 修改已审核凭证时，需要：反审核 → 修改 → 重新审核
- 操作步骤繁琐，容易出错

**优化建议**:

- 提供"修改并重新审核"功能
- 后端自动处理：检查权限 → 反审核 → 修改 → 审核
- 记录操作日志，标注"修改后重新审核"

### 五、审核/反审核流程

#### 问题 11：审核人不能是制单人，但没有前置检查

**位置**: `server/src/services/voucherEntry.ts:validateVoucherForAudit`

**问题描述**:

- 只在提交审核时检查
- 如果审核人就是制单人，应该在进入审核页面时就提示

**优化建议**:

- 前端在显示"审核"按钮前调用，如果不能审核则禁用按钮并显示原因

#### 问题 12：批量审核功能缺失

**位置**: `server/src/routes/voucherAudit.ts`

**问题描述**:

- 只支持单个凭证审核
- 如果一个月有100张凭证，需要点击100次

**优化建议**:

```typescript
// 增加批量审核接口
router.post(
  '/vouchers/batch-audit',
  operationLog('批量审核', '凭证管理'),
  (req: AuthRequest, res) => {
    const { voucherIds } = req.body
    if (!Array.isArray(voucherIds) || voucherIds.length === 0) {
      return res.status(400).json({ code: 400, message: '请选择要审核的凭证' })
    }

    const results = []
    for (const id of voucherIds) {
      try {
        // 审核逻辑
        results.push({ id, success: true })
      } catch (error) {
        results.push({ id, success: false, error: error.message })
      }
    }

    res.json({
      code: 0,
      message: `批量审核完成，成功${results.filter(r => r.success).length}条`,
      data: results,
    })
  }
)
```

### 六、过账/反过账流程

#### 问题 13：过账前没有数据完整性检查

**位置**: `server/src/routes/voucherPosting.ts:20-60`

**问题描述**:

- 只检查了审核状态
- 没有检查：
  - 科目余额是否会出现负数（对于不允许负数的科目）
  - 辅助核算项目是否完整
  - 是否有未保存的修改

**优化建议**:

```typescript
// 过账前完整性检查
const integrityErrors = []

// 检查科目余额
for (const entry of entries) {
  const account = db.prepare('SELECT * FROM accounts WHERE id = ?').get(entry.account_id)
  if (account.no_negative === 0) {
    // 不允许负数
    const balance = calculateBalanceAfterPosting(entry)
    if (balance < 0) {
      integrityErrors.push(`科目${account.name}余额将为负数`)
    }
  }
}

// 检查辅助核算
for (const entry of entries) {
  const account = db.prepare('SELECT * FROM accounts WHERE id = ?').get(entry.account_id)
  if (account.is_aux === 1) {
    const auxTypes = JSON.parse(account.aux_types || '{}')
    for (const auxType of Object.keys(auxTypes)) {
      if (!entry[`${auxType}_id`]) {
        integrityErrors.push(`科目${account.name}缺少${auxType}辅助核算`)
      }
    }
  }
}

if (integrityErrors.length > 0) {
  return res.status(400).json({
    code: 400,
    message: '过账前检查失败',
    data: { errors: integrityErrors },
  })
}
```

### 九、通用问题

#### 问题 20：操作日志记录不够详细

**位置**: 所有使用 `operationLog` 的地方

**问题描述**:

- 只记录了操作类型和模块
- 没有记录操作前后的数据变化

**优化建议**:

```typescript
// 增强操作日志
export function operationLog(action: string, module: string) {
  return (req: AuthRequest, res: Response, next: NextFunction) => {
    const originalJson = res.json.bind(res)

    res.json = function (data: any) {
      // 记录详细日志
      db.prepare(
        `
        INSERT INTO operation_logs 
        (id, user_id, action, module, request_data, response_data, ip, created_at)
        VALUES (?, ?, ?, ?, ?, ?, ?, datetime('now'))
      `
      ).run(
        uuidv4(),
        req.userId,
        action,
        module,
        JSON.stringify(req.body),
        JSON.stringify(data),
        req.ip
      )

      return originalJson(data)
    }

    next()
  }
}
```

## 总结

通过代码审查发现了 **22 个操作层面的问题**，主要集中在：

1. **缺少操作引导**：用户不知道下一步该做什么
2. **缺少批量操作**：重复操作效率低
3. **错误提示不友好**：不知道如何解决问题
4. **缺少数据校验**：可能导致数据不一致
5. **缺少操作确认**：容易误操作

建议按优先级逐步优化，优先解决影响数据准确性的问题。
